# Documentation Guidelines

Good documentation makes Zephyr accessible to everyone. This guide covers contributing to code comments, README files, and the documentation site.

> **Confession time:** We've all skipped writing docs at some point, thinking "I'll remember what this does." Spoiler alert: you won't! Future-you (and your teammates) will thank you for taking a few extra minutes to document your code. Plus, writing good docs often helps you spot bugs and design issues before they become problems! üìù

## Why Documentation Matters

- **Onboarding**: Helps new contributors get started quickly
- **Maintenance**: Makes code easier to understand and modify
- **Discoverability**: Helps users find features they need
- **Professionalism**: Shows care and attention to detail

*Think of documentation as a love letter to future developers (who might be you in 6 months, confused about what past-you was thinking!).*

## Code Documentation

### JSDoc Comments

Use JSDoc for all public APIs:

```typescript
/**
 * Fetches a user's profile from the database.
 *
 * @param userId - The unique identifier of the user
 * @param options - Optional fetch configuration
 * @returns The user profile or null if not found
 * @throws {DatabaseError} If database connection fails
 *
 * @example
 * ```ts
 * const profile = await getUserProfile("user-123");
 * if (profile) {
 *   console.log(profile.displayName);
 * }
 * ```
 */
export async function getUserProfile(
  userId: string,
  options?: FetchOptions
): Promise<UserProfile | null> {
  // Implementation...
}
```

**JSDoc Tags to Use:**
- `@param` - Parameter description
- `@returns` - Return value description
- `@throws` - Exceptions that may be thrown
- `@example` - Usage examples
- `@deprecated` - Mark deprecated APIs
- `@see` - Link to related documentation
- `@since` - Version when feature was added

### Inline Comments

Comment complex logic, not obvious code:

```typescript
// ‚ùå Bad - obvious comment
// Increment counter by 1
counter++;

// ‚úÖ Good - explains "why"
// Debounce search to avoid excessive API calls during typing
const debouncedSearch = useDebouncedCallback(search, 300);

// ‚úÖ Good - explains complex algorithm
// Calculate Aura score using weighted factors:
// - Post engagement: 40%
// - Account age: 30%
// - Community contributions: 30%
const auraScore = calculateWeightedScore(metrics);
```

## README Documentation

### Project README

When updating the main README (`README.md` or `.github/README.md`):

**Structure:**
1. Project banner/logo
2. Brief description
3. Key features
4. Quick start guide
5. Documentation links
6. Contributing guide
7. License information

**Best Practices:**
- Keep it concise (scannable in 2-3 minutes)
- Use visuals (screenshots, diagrams)
- Provide working code examples
- Link to detailed docs
- Keep up-to-date

### Content Guidelines

**DO ‚úÖ**
- Use clear, concise language
- Include code examples
- Add screenshots for UI features
- Link to related docs

## Documentation Types

**Purpose**: Teach users how to accomplish a specific task

**Structure:**
1. What you'll build
2. Prerequisites
3. Step-by-step instructions
4. Verification/testing
5. Next steps

**Example:**
```mdx

## What You'll Build

A simple post with text content and an image attachment.

## Prerequisites

- Zephyr account
- Basic knowledge of the platform

## Steps

### 1. Navigate to Post Creator
...

### 2. Write Your Content

```

### How-To Guides

**Purpose**: Solve specific problems

**Structure:**
1. Problem statement
2. Solution overview
3. Detailed steps
4. Troubleshooting

**Example:**
```mdx
# How to Enable Dark Mode

## Problem

You want to switch the UI to dark mode.

## Solution

Navigate to Settings > Appearance > Theme and select "Dark".

## Steps

1. Click your avatar in the top right
2. Select "Settings"
3. Go to "Appearance"
4. Click "Dark" under Theme
5. Changes apply immediately

## Troubleshooting

**Theme not changing?**
- Clear browser cache
- Check browser compatibility
```

### Reference Documentation

**Purpose**: Provide technical specifications

**Structure:**
1. Overview
2. API/Component details
3. Parameters/props
4. Return values
5. Examples

**Example:**
```mdx
# API Reference: createPost

Creates a new post in the database.

## Signature

\`\`\`typescript
function createPost(data: PostData): Promise<Post>
\`\`\`

## Parameters

- `data` (PostData): Post content and metadata
  - `content` (string): Post text (max 5000 chars)
  - `media` (MediaAttachment[]): Optional attachments
  - `visibility` (Visibility): Post visibility

## Returns

Promise<Post>: The created post object

## Throws

- `ValidationError`: Invalid post data
- `DatabaseError`: Database operation failed

## Examples

\`\`\`typescript
const post = await createPost({
  content: "Hello, world!",
  visibility: "public"
});
\`\`\`
```

**Purpose**: Explain concepts and design decisions

**Structure:**
1. Concept introduction
2. Why it matters
3. How it works
4. When to use
5. Related concepts

As a user, higher Aura unlocks:

- Enhanced features

## Style Guide

### Voice and Tone

- **Be conversational**: "You can configure this..." not "One can configure this..."
- **Be direct**: "Click the button" not "The button should be clicked"
- **Be positive**: "Use this approach" not "Don't use that approach"
- **Be inclusive**: "they/them" for generic examples

### Formatting

- **Headings**: Sentence case (not Title Case)
- **Lists**: Parallel structure (all start with verb or all nouns)
- **Code**: Backticks for `inline code`, blocks for multi-line
- **Emphasis**: **Bold** for UI elements, _italic_ for emphasis

### Examples

```mdx
# ‚úÖ Good

To create a post:

1. Click the **New Post** button
2. Enter your content
3. Click **Publish**

# ‚ùå Bad

How To Create A Post:

1. The user should click on the "New Post" button
2. Then they can enter content
3. Finally, clicking "Publish" will publish it
```

## Keeping Docs Updated

### When Code Changes

- Update relevant docs in the same PR
- Mark deprecated features
- Add migration guides for breaking changes

### Regular Maintenance

- Review docs quarterly
- Remove outdated content
- Update screenshots
- Fix broken links
- Improve clarity based on user feedback

> **Documentation truth bomb:** Outdated docs are worse than no docs at all! They mislead people and waste time. When you change code, updating the docs isn't "extra work" - it's part of the feature. Your fellow developers (and users!) will appreciate the accuracy. üíØ
